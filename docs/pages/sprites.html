Sprites

Add visual life to your scenario with sprite sheets for units, buildings, biomes, resources, and effects.

What Is a Sprite Sheet?

A sprite sheet is a single PNG image that packs multiple frames into a grid. Each row represents a different animation (idle, move, attack, etc.) and each column represents a single frame within that animation. The engine slices the image at runtime using the frame dimensions you provide.

Format

All sprite files must be PNG with a transparent background. The engine does not support JPEG, GIF, or WebP for sprite sheets. Transparency is required so sprites composite cleanly over the hex grid and other layers.

How Animations Work

Animations are configured with five properties:

PropertyTypeDescription
frameWnumberWidth of a single frame in pixels. Defines the column grid.
frameHnumberHeight of a single frame in pixels. Defines the row grid.
rownumberWhich row of the sprite sheet to use for this animation (0-indexed).
framesnumberNumber of columns (frames) in the animation.
fpsnumberPlayback speed in frames per second.
loopbooleanWhether the animation repeats. Set to false for one-shot animations like death.

Unit Sprites

Units support multiple animation types. Each animation is a named key inside the unit's sprite block:

AnimationWhen Played
idleUnit is standing still, waiting for orders.
moveUnit is traveling between hexes.
attackUnit performs a melee or ranged attack.
dieUnit is destroyed (typically loop: false).
customAny additional name you define, triggered via abilities or events.
 
s.unit("knight", {
  hp: 80, attack: 18, defense: 12,
  cost: { gold: 40 },
  sprite: {
    file: "knight.png",
    frameW: 64,
    frameH: 64,
    animations: {
      idle:   { row: 0, frames: 4, fps: 4, loop: true },
      move:   { row: 1, frames: 6, fps: 10, loop: true },
      attack: { row: 2, frames: 6, fps: 12, loop: false },
      die:    { row: 3, frames: 4, fps: 8, loop: false }
    }
  }
});

Building Sprites

Buildings can be static or animated.

Static buildings

A static building uses a single image file with optional damage state variants:

s.building("castle", {
  hp: 500,
  sprite: {
    file: "castle.png",
    scale: 1.2,
    damaged: "castle_damaged.png",
    ruined: "castle_ruined.png"
  }
});

Building damage states

StatePropertyCondition
NormalfileHP > 50%
DamageddamagedHP < 50%
RuinedruinedHP < 20%

If you omit damaged and ruined, the engine will tint the normal sprite darker as health decreases.

Animated buildings

Animated buildings use the same frameW, frameH, frames, fps, and loop properties as units, but typically only have a single animation row.

Biome Tiles

Biome sprites fill the hex grid. They can be static (a single image tile) or animated (for water ripples, lava glow, etc.):

s.biome("water", {
  color: "#2196f3",
  moveCost: 3,
  sprite: {
    file: "water.png",
    frameW: 64,
    frameH: 64,
    frames: 8,
    fps: 4,
    loop: true
  }
});

Resource Sprites

Resources use two separate images:

  • icon — shown in the UI resource panel (small, typically 32x32).
  • sprite — shown on the map tile where the resource deposit sits.
s.resource("gold", {
  color: "#ffd700",
  rarity: "common",
  icon: "gold_icon.png",
  sprite: "gold_tile.png"
});

Effect Sprites

Visual effects (explosions, healing bursts, spell impacts) are animated sprites triggered by abilities or events:

s.effect("fireball_impact", {
  file: "fireball_impact.png",
  frameW: 128,
  frameH: 128,
  frames: 8,
  fps: 16,
  loop: false
});

Fallback Rendering

If no sprite is defined for a unit, building, biome, or resource, the engine falls back to colored hexagons. The color is taken from the color field in each definition. This means every scenario works visually even without any sprite assets — you can always add graphics later.

The color field is always required, even when sprites are provided. It is used for the minimap, selection highlights, and accessibility modes.

Scale Parameter

The scale property controls how large the sprite renders relative to the hex cell. The default is 1.0, which fits the sprite exactly within the hex. Use values greater than 1.0 for sprites that should overflow (e.g., large buildings or boss units), or less than 1.0 for smaller elements.

sprite: {
  file: "dragon.png",
  scale: 1.5,    // 50% larger than a standard hex
  // ...
}

Sprite Folder Structure

Place all sprite files in a sprites/ directory inside your scenario folder:

my_scenario/sprites/
├── knight.png
├── archer.png
├── castle.png
├── castle_damaged.png
├── plains.png
└── gold_icon.png

All file paths in the DSL are relative to this sprites/ directory. You do not need to include the directory prefix in your definitions.

Full Example

Here is a complete s.sprites() configuration from the Castle scenario:

s.sprites({
  units: {
    knight: {
      file: "knight.png",
      frameW: 64, frameH: 64,
      animations: {
        idle:   { row: 0, frames: 4, fps: 4, loop: true },
        move:   { row: 1, frames: 6, fps: 10, loop: true },
        attack: { row: 2, frames: 6, fps: 12, loop: false },
        die:    { row: 3, frames: 4, fps: 8, loop: false }
      }
    },
    archer: {
      file: "archer.png",
      frameW: 64, frameH: 64,
      animations: {
        idle:   { row: 0, frames: 4, fps: 4, loop: true },
        move:   { row: 1, frames: 6, fps: 10, loop: true },
        attack: { row: 2, frames: 8, fps: 14, loop: false },
        die:    { row: 3, frames: 4, fps: 8, loop: false }
      }
    },
    dragon: {
      file: "dragon.png",
      frameW: 128, frameH: 128,
      scale: 1.5,
      animations: {
        idle:   { row: 0, frames: 6, fps: 6, loop: true },
        move:   { row: 1, frames: 8, fps: 10, loop: true },
        attack: { row: 2, frames: 10, fps: 14, loop: false },
        die:    { row: 3, frames: 6, fps: 8, loop: false }
      }
    }
  },
  buildings: {
    castle:   { file: "castle.png", scale: 1.2,
                damaged: "castle_damaged.png",
                ruined: "castle_ruined.png" },
    barracks: { file: "barracks.png" },
    tower:    { file: "tower.png" }
  },
  biomes: {
    plains: { file: "plains.png" },
    forest: { file: "forest.png" },
    water:  { file: "water.png", frameW: 64, frameH: 64,
              frames: 8, fps: 4, loop: true },
    lava:   { file: "lava.png", frameW: 64, frameH: 64,
              frames: 6, fps: 3, loop: true }
  },
  resources: {
    gold:  { icon: "gold_icon.png",  sprite: "gold_tile.png" },
    wood:  { icon: "wood_icon.png",  sprite: "wood_tile.png" },
    stone: { icon: "stone_icon.png", sprite: "stone_tile.png" }
  },
  effects: {
    fireball_impact: { file: "fireball_impact.png",
                       frameW: 128, frameH: 128,
                       frames: 8, fps: 16, loop: false },
    heal_burst:      { file: "heal_burst.png",
                       frameW: 64, frameH: 64,
                       frames: 6, fps: 12, loop: false }
  }
});

Where to Get Free Sprites

You do not need to be an artist. These sites offer high-quality free assets:

kenney.nl

Professional-quality asset packs, all CC0 (public domain). Hex tiles, characters, UI elements, and more.

opengameart.org

Large community library. Look for CC0 or CC-BY licensed sprite sheets. Always check the license per asset.

itch.io/game-assets

Many creators publish free asset packs. Filter by "free" and check the license before using.

AI Sprite Generation

If you need custom sprites and cannot draw them yourself, AI tools can help:

PixelLab.ai

Best option for pixel art and sprite sheet animations. Generates consistent multi-frame sheets from text prompts.

Ludo.ai

Text-to-sprite-to-spritesheet pipeline. Good for rapid prototyping of unit and building art.

PixelBox

Free, no registration required. Simple pixel art generator for quick placeholder sprites.

Start with colored hexagons (no sprites at all) to get your gameplay right, then add visuals once you are happy with the balance. Art is optional — the DSL works perfectly without it.
